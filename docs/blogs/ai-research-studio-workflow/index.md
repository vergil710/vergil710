我越来越觉得，AI 辅助科研的关键不是“问一个很聪明的问题”，而是把 AI 放进稳定的工作流里。

零散聊天当然有用，但它很难积累。真正有长期价值的是：每次讨论、每次推导、每次代码修改、每次文献阅读，都能沉淀到一个项目结构里。

## 1. 按场景组织，而不是按模型组织

不同模型擅长的事情不一样，但研究任务本身更应该按场景组织：

- 控制理论讨论班；
- 论文写作工厂；
- 代码与仿真实验；
- 文献卡片和知识库；
- 哲学/方法论讨论。

模型只是工具，场景才是长期记忆的容器。

## 2. 一个任务只设一个主模型

多模型协作很容易变成意见堆叠。更合理的方式是：

- 一个任务只设一个主模型；
- 其他模型只做复核、批评或补充；
- 重要结论必须回到文献、实验或人工判断。

这样可以减少互相打架的建议，也更容易追踪责任边界。

## 3. Skill 化

好用的 prompt 不应该只停留在聊天记录里。它可以被整理成 Skill：

- 输入是什么；
- 输出是什么；
- 必须遵守哪些约束；
- 需要人工确认哪些点；
- 结束后应该更新哪些文件。

当一个任务被 Skill 化，它就从“临时技巧”变成了“可复用流程”。

## 4. 日志化

科研最怕的是过一段时间之后忘记当时为什么这么做。

所以重要任务结束后，至少要留下三类记录：

- 改了什么；
- 为什么改；
- 还有什么没完成。

这比一次性追求完美更重要。长期项目不是靠记忆维持的，而是靠结构和记录维持的。
