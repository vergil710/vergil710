MPC 的直觉很简单：每一步都向前看一段时间，解一个有限时域优化问题，只执行第一个控制量，然后滚动更新。

但把它放到列车控制里，难点不在“会不会写优化问题”，而在模型、约束、实时性和工程可用性。

## 1. 预测模型

列车纵向动力学可以写成：

\\[
m\\dot v = F_t(u_t, v) - F_b(u_b, v) - R(v) - mg\\sin\\theta(s)
\\]

这里至少涉及牵引力、制动力、基本阻力、坡度阻力和位置相关线路条件。模型越真实，优化越难；模型越简化，控制效果越可能偏离实际。

## 2. 目标函数

一个典型目标会同时考虑：

- 到站时间误差；
- 牵引能耗；
- 速度跟踪误差；
- 控制量变化率；
- 舒适度或冲击率。

这些目标之间不一定一致。节能可能倾向惰行，准点可能需要更积极牵引，舒适度又会限制控制变化。

## 3. 约束

列车控制中的约束不是装饰项：

- 速度不能超过限速；
- 位置必须满足移动授权；
- 牵引和制动有幅值限制；
- 控制等级可能是离散的；
- 分相区有特殊通过规则；
- 计算时间不能超限。

如果约束不可行，系统还需要有降级策略，而不是简单宣告求解失败。

## 4. 实时性

MPC 在论文里经常默认求解器足够快，但车载平台上的计算资源有限。实际部署时，必须回答：

- 单周期是否能完成求解；
- 求解失败时输出什么；
- 是否可以 warm start；
- 是否可以降低预测步长或减少迭代次数；
- 优化器运行时间如何被监控。

MPC 的价值在于它能自然处理多约束问题；它的风险也在于，真实约束一多，计算复杂度和工程调参会迅速上升。
